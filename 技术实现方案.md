## 二、技术选型决策分析

### 2.1 为什么用 SSE 而不是 WebSocket？

**场景分析**：AI 生成是典型的"请求-流式响应"单向推送模式，客户端发一次请求，服务端持续推送文本 chunk。

| 方案 | 优点 | 缺点 |
|------|------|------|
| WebSocket | 双向通信 | 需要连接管理、心跳、重连逻辑，对此场景过度设计 |
| SSE (EventSource) | 浏览器原生支持、自动重连 | **只支持 GET 请求**，无法携带 JSON body |
| SSE (fetch 手动解析) | 支持 POST + JSON body，轻量 | 需要自己写流解析逻辑 |

本项目选择了**第三种**：用 `fetch()` 发送 POST 请求，通过 `ReadableStream` 手动解析 SSE 格式。这是 AI 应用中最常见的做法（ChatGPT 也是这个模式）。

**关键实现**（`frontend/src/services/sse.ts`）：
```typescript
// 用 fetch 发 POST 请求，而非浏览器原生 EventSource（只支持 GET）
fetch('/api/chat/stream', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ messages, gitData, isInitial }),
  signal: abortController.signal,  // 支持取消
})
.then(async (response) => {
  const reader = response.body?.getReader();
  const decoder = new TextDecoder();
  let buffer = ''; // 处理不完整的数据行

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;

    // 追加到缓冲区，按行分割处理
    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split('\n');
    buffer = lines.pop() || ''; // 保留不完整的行

    for (const line of lines) {
      if (line.startsWith('data: ')) {
        const data = JSON.parse(line.slice(6));
        // data.chunk / data.done / data.error 三种消息类型
      }
    }
  }
});
```

### 2.2 为什么用 Valtio 而不是 Redux/Zustand？

Valtio 的核心优势是**直接修改即响应**，用 `proxy()` 包裹的对象可以直接赋值，Valtio 自动追踪变化触发重渲染。

对比：
```typescript
// Redux: 需要 dispatch + action + reducer
dispatch({ type: 'APPEND_CHUNK', payload: { id, chunk } });

// Zustand: 需要 set 函数
set(state => ({ messages: state.messages.map(...) }));

// Valtio: 直接修改（本项目的做法）
const message = chatStore.messages.find(m => m.id === messageId);
message.content += chunk;  // 直接赋值，自动触发 UI 更新
```

在流式场景下，每秒可能有几十次 chunk 追加操作，Valtio 的写法最简洁。

### 2.3 为什么用 Rspack 而不是 Webpack/Vite？

Rspack 是 Rust 实现的 Webpack 兼容构建工具，既保留了 Webpack 生态（loader/plugin 兼容），又有接近 Vite 的速度。适合需要 Webpack 配置能力但又追求速度的场景。

## 三、核心实现细节

### 3.1 Git 数据采集层

**文件**：`backend/src/services/gitAnalyzer.ts`

**实现思路**：用 `simple-git` 库封装 Git 命令，分三步完成数据采集：

**第一步：获取 commit 列表**
```typescript
const logOptions = {
  '--since': since,    // 起始日期
  '--until': until,    // 结束日期
  '--no-merges': null, // 排除 merge commit（通常无实际内容）
};
if (author) logOptions['--author'] = author;
const log = await git.log(logOptions);
```

**第二步：逐 commit 获取文件级变更**
```typescript
// git show --stat 获取每个 commit 改了哪些文件、增删多少行
const showResult = await git.show([commit.hash, '--stat', '--format=']);
// 正则解析 " src/main.ts | 10 +++---" 格式
const match = line.match(/^\s*([^|]+)\|\s*([\d]+)\s+([+\-]+)$/);
```

**第三步：按 commit message 关键词分类**

这里采用了**规则匹配**而非 AI 分类，原因是：
- Git commit message 有约定俗成的格式（Conventional Commits）
- 规则匹配速度快、确定性高、不消耗 AI 额度
- 支持中英文关键词双语匹配

```typescript
// 前缀匹配 + 关键词包含，同时覆盖中英文
if (msg.startsWith('feat') || msg.includes('新增') || msg.includes('添加')) {
  categories.feature.push(commit);
}
```

**设计启示**：Agent 不是所有环节都要用 AI，对于结构化、确定性高的任务（如分类），规则引擎比 AI 更可靠。

**数据格式化**：将结构化的 Git 分析结果转换为 AI 能理解的文本：

private formatGitData(gitData: GitAnalysisResult): string {

### 3.3 模型降级与流式输出

**文件**：`backend/src/services/aiClient.ts`

这是整个系统最关键的模块，解决了免费模型不稳定的核心痛点。

**当前模型降级队列**（2024年最新配置）：

**降级策略设计**：

```typescript
async streamWithFallback(
  options: AIStreamOptions,
  onChunk: (chunk: string) => void,
): Promise<void> {
  // 9 个模型组成降级队列
  const models = [options.model || this.defaultModel, ...this.backupModels];
  const errors: string[] = [];

  for (const model of models) {
    try {
      console.log(`[AI] 尝试模型: ${model}`);
      const result = this.createStream({ ...options, model });
      let hasOutput = false;

      for await (const chunk of result.textStream) {
        hasOutput = true;
        onChunk(chunk);  // 通过回调实时推送每个 chunk
      }

      // 空内容检测：有的模型不报错但返回空，也要降级
      if (!hasOutput) {
        errors.push(`${model}: 返回空内容`);
        continue;
      }

      return; // 成功则直接返回
    } catch (error) {
      errors.push(`${model}: ${error.message}`);
      continue; // 失败则尝试下一个模型
    }
  }

  throw new Error(`所有模型都不可用:\n${errors.join('\n')}`);
}
```

**为什么用回调而不是 async generator？**

早期版本使用了嵌套 async generator 模式（`yield* innerGenerator()`），发现两个问题：
1. 错误无法正确从内层 generator 传播到外层
2. generator 的 `return()` 调用时序在嵌套场景下不可控

回调模式更简单直接：每产生一个 chunk 就调用 `onChunk()`，调用方（路由层）直接写入 SSE 响应。

**`maxRetries: 0` 的关键性**：

```typescript
return streamText({
  model: this.openrouter(model),
  messages: messages as any,
  maxRetries: 0, // 禁用 SDK 内置重试
});
```

Vercel AI SDK 默认 `maxRetries: 2`（即最多 3 次尝试），每次使用指数退避。当一个模型持续 429 时：
- 第 1 次：立即失败
- 第 2 次：等 1 秒后重试
- 第 3 次：等 2 秒后重试

9 个模型全部 429 的场景下：`9 模型 × 3 次 × 平均 1.5 秒 = 40+ 秒`。用户会感知到系统"卡死"。

禁用 SDK 重试后，由应用层控制降级，每个模型只试一次，失败立即切换，所有模型失败也只需几秒。

### 3.4 前端打字机效果

**文件**：`frontend/src/services/sse.ts`

**问题背景**：免费模型通常不支持真正的流式输出，会一次性返回大块文本（如一次返回 280 字符），导致没有打字机效果。

**解决方案**：前端模拟打字机效果，即使后端一次性返回大块文本，也逐字显示。

```typescript
// 打字机效果队列
class TypewriterQueue {
  private queue: string[] = [];
  private isProcessing = false;
  private callback: (char: string) => void;
  private charDelay: number;

  constructor(callback: (char: string) => void, charDelay = 12) {
    this.callback = callback;
    this.charDelay = charDelay; // 12ms/字符 ≈ 80字/秒
  }

  // 添加文本到队列
  enqueue(text: string) {
    for (const char of text) {
      this.queue.push(char);
    }
    this.process();
  }

  // 处理队列，逐字输出
  private async process() {
    if (this.isProcessing) return;
    this.isProcessing = true;

    while (this.queue.length > 0) {
      const char = this.queue.shift()!;
      this.callback(char);
      
      // 根据字符类型调整延迟
      let delay = this.charDelay;
      if (char === '\n') {
        delay = this.charDelay * 2; // 换行稍慢
      } else if (/[，。！？、；：]/.test(char)) {
        delay = this.charDelay * 1.5; // 标点稍慢
      }
      await new Promise(r => setTimeout(r, delay));
    }

    this.isProcessing = false;
  }

  // 立即输出剩余内容（流结束时调用）
  flush() {
    if (this.queue.length > 0) {
      this.callback(this.queue.join(''));
      this.queue = [];
    }
    this.isProcessing = false;
  }
}
```

**使用方式**：

```typescript
export async function sendConversation(message: string, context?: any): Promise<void> {
  const messageId = chatActions.startAssistantMessage();

  // 创建打字机队列，每个字符通过回调更新 store
  const typewriter = new TypewriterQueue((char) => {
    chatActions.appendAssistantContent(messageId, char);
  }, 12); // 12ms/字符

  const disconnect = createConversationConnection({
    message,
    context,
    onChunk: (chunk) => {
      // 将后端返回的大块文本加入队列，逐字输出
      typewriter.enqueue(chunk);
    },
    onComplete: () => {
      setTimeout(() => {
        typewriter.flush(); // 确保所有内容都显示
        chatActions.finishAssistantMessage(messageId);
      }, 100);
    },
  });
}
```

### 3.5 SSE 服务端实现

**文件**：`backend/src/routes/chat.ts`

SSE 不只是设置 `Content-Type: text/event-stream` 就完了，生产环境需要考虑几个工程问题：

**问题 1：Nginx/代理缓冲**

```typescript
res.setHeader('X-Accel-Buffering', 'no');
res.setHeader('Content-Encoding', 'identity'); // 明确禁用压缩
```

Nginx 默认会缓冲后端响应到一定大小才发给客户端，这会导致流式效果变成"一坨一坨"输出。

**问题 2：空闲连接被断开**

```typescript
// 每 15 秒发送心跳
const heartbeat = setInterval(() => {
  res.write(': heartbeat\n\n');  // SSE 注释格式，客户端不会解析为数据
}, 15_000);
```

`:` 开头的行是 SSE 协议的注释，客户端会忽略但网络层会认为连接是活跃的。

**问题 3：无限等待**

```typescript
// 60 秒超时保护
const timeout = setTimeout(() => {
  res.write(`data: ${JSON.stringify({ error: '生成超时，请重试' })}\n\n`);
  res.end();
}, STREAM_TIMEOUT);
```

**问题 4：资源泄露**

```typescript
// 客户端断开时清理定时器
req.on('close', () => {
  clearInterval(heartbeat);
  clearTimeout(timeout);
});

// 正常结束也要清理
finally {
  clearInterval(heartbeat);
  clearTimeout(timeout);
  res.end();
}
```

**问题 5：强制刷新缓冲区**

```typescript
const sendSSE = (data: object) => {
  const line = `data: ${JSON.stringify(data)}\n\n`;
  res.write(line);
  // 强制刷新缓冲区
  if (typeof (res as any).flush === 'function') {
    (res as any).flush();
  }
};
```

### 3.6 前端状态管理与流式更新

**文件**：`frontend/src/stores/chatStore.ts`

流式 AI 应用的状态管理核心挑战：**高频更新**。每个 chunk 到达都要更新 UI。

**状态设计**：

```typescript
export const chatStore = proxy<ChatState>({
  messages: [],         // 消息列表
  isStreaming: boolean, // 全局流式状态（控制 UI 交互禁用）
  gitConfig: null,      // Git 配置
  gitData: null,        // Git 分析结果
  error: null,          // 错误信息
});
```

**流式消息的生命周期**：

```
startAssistantMessage()    → 创建空消息占位 { content: '', isStreaming: true }
    ↓
appendAssistantContent()   → 逐字符追加 message.content += char（高频调用）
    ↓
finishAssistantMessage()   → 标记完成 { isStreaming: false }
```

这种"先占位、再追加、后完成"的三阶段模式是流式 AI 应用的标准做法。

## 五、可复用的设计模式

### 模式 1：SSE + fetch 替代 EventSource

适用场景：需要 POST 请求体的流式接口。几乎所有 AI 聊天应用都可以复用这个模式。

### 模式 2：模型降级队列

适用场景：依赖外部 API 且可用性不稳定的系统。不仅限于 AI 模型，任何多供应商场景（CDN、支付、短信）都可以用这个模式。

### 模式 3：占位-追加-完成的流式状态管理

适用场景：任何需要流式更新 UI 的场景（AI 对话、实时日志、文件上传进度）。

### 模式 4：规则引擎 + AI 混合处理

适用场景：Agent 系统中，确定性高的环节用规则，不确定性高的环节用 AI。本项目中 Git 分类用规则，周报撰写用 AI。

### 模式 5：Stateless 后端 + 前端管理对话历史

适用场景：需要水平扩展的 AI 对话系统。后端不存储状态，前端每次发送完整 messages，后端任意节点都能处理。

### 模式 6：前端打字机效果模拟

适用场景：当后端 API 不支持真正流式输出（一次返回大块文本）时，前端通过队列实现逐字显示效果。

```typescript
// 核心思路：将大块文本拆分为字符队列，定时输出
class TypewriterQueue {
  enqueue(text) { /* 拆分入队 */ }
  process()     { /* 定时出队 */ }
  flush()       { /* 立即清空 */ }
}
```

## 六、UI/UX 设计要点

### 6.1 响应式布局

## 七、潜在优化方向

1. **消息历史截断** — 随着对话轮次增多，messages 数组不断增大。可以限制最近 N 轮或用 token 计数截断
2. **流式中断恢复** — 当前中断后只能重新生成。可以保存已生成部分，从断点继续
3. **Git 分析缓存** — 同一仓库、同一时间范围的分析结果可以缓存，避免重复执行 git 命令
4. **并发 commit 解析** — 当前逐个 commit 串行调用 `git show`，可以用 `Promise.all` 并发（注意控制并发数）
5. **Prompt 优化** — 可以根据 commit 数量动态调整提示词，少量 commit 时要求更详细，大量时要求更精炼
6. **打字机速度自适应** — 根据文本长度动态调整速度，长文本加快，短文本保持节奏
